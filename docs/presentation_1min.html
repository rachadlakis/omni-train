Why Do We Need Distributed Training?
Scaling from a notebook to real-world models requires more than one GPU
1 / 5
📈
Bigger = Better
The best-performing models are the largest ones. GPT-5.4, Gemini 3.1 Pro, Llama 4 — scale is the single most reliable way to improve accuracy, reasoning, and generalization.
70B+params in top models
⚠️
PEFT & Quantization Have Limits
LoRA, QLoRA, and 4-bit quantization reduce VRAM but come at a cost — lower model capacity, degraded accuracy on complex tasks, and constraints on which layers you can update. Full fine-tune always wins on quality.
⏱️
One GPU Does Not Fit the Required VRAM — Even If It Did, a Single GPU Is Too Slow
Full fine-tuning requires ~12× the model's bf16 size in VRAM:
  • Weights (bf16) — 2 bytes/param
  • Gradients (bf16) — 2 bytes/param
  • AdamW m₁ + m₂ (fp32) — 8 bytes/param

LLaMA-2 7B = 14 GB weights → 84 GB total. An A100 80 GB can't hold it. And even if it could, training at scale takes days — distributing across 8 GPUs cuts that to hours.

12×model size in VRAM

The Problem — What You Need to Know Before You Start
Distributed training demands expertise across many layers — all before writing a single line of training code

🔗
Communication Primitives
  • torchrun — RANK, LOCAL_RANK, WORLD_SIZE, init_process_group, rendezvous & elastic launch
  • NCCL Collectives — AllReduce, ReduceScatter, AllGather, Broadcast, Barrier
  • DDP — gradient all-reduce sync, bucket overlap, no_sync context, rank-0 writes
  • ZeRO — stage 1/2/3 optimizer & gradient sharding, how FSDP maps to ZeRO-3
⚙️
Sharding & Memory
  • FSDP fully_shard — wrap policy, meta-device init, leaf-first order, all-gather / reduce-scatter
  • Mixed precision & dtypes — param / reduce / output dtype interaction, bf16 vs fp32 optimizer states
  • Prefetching — forward & backward prefetch depth, overlapping all-gather with compute
  • Gradient checkpointing — recompute activations to trade compute for memory
💾
Checkpointing & Libraries
  • DCP / DTensor — sharded save & load, broadcast_from_rank0, state-dict reassembly
  • Saving / loading sharded models — rank-consistent reassembly across strategies
  • Libraries — PyTorch, Transformers, BitsAndBytes, PEFT, torchvision — each with its own distributed quirks
  • Silent failures — deadlocks, NaN loss, gradient corruption — often with no error message

What Omni-Train Does For You
  • FSDP2 sharding — meta-device init, leaf-first wrap order, all-gather / reduce-scatter, mixed-precision policy
  • DDP — per-rank model load, all-reduce gradient sync, rank-0 checkpoint write
  • LoRA / QLoRA — PEFT adapter injection, 4-bit BitsAndBytes nf4/fp4, k-bit training prep, optional double quantization
  • PEFT + FSDP — LoRA adapters correctly wrapped inside FSDP sharding boundaries; gradients flow through sharded adapters without corruption
  • No setup code needed — GPU assignment, process coordination, and model loading across ranks are all taken care of automatically
  • Mixed precision — bf16 params, fp32 optimizer states, configurable reduce/output dtypes — no manual casting
  • W&B monitoring — opt-in loss curves, GPU memory snapshots, and per-rank diagnostics via all_gather_object
  • 20+ architectures — LLMs, Vision CNNs, YOLO, VLMs, embedding models — one model_type field
  • Single → multi-GPU — switch from solo to DDP to FSDP by changing one line in the YAML, zero code changes
Web UI / CLI + YAML File
Option 1 — CLI: edit config.yaml, then launch:
# single GPU python train.py --config config.yaml # multi-GPU (torchrun) bash launch.sh # SLURM cluster sbatch scripts/slurm_train.sh

Option 2 — Web UI: bash ui/launch_ui.sh → visual YAML editor. Toggle LoRA, pick strategy, click Launch — live logs stream in the browser.

SLURM — Multi-Node Training
Today torchrun spawns processes on one node. SLURM spans them across many nodes, each with its own GPUs, over InfiniBand.
  • Auto-generated job scripts from config.yaml — MASTER_ADDR, MASTER_PORT, WORLD_SIZE set automatically
  • Elastic rendezvous via torchrun --nnodes=N --rdzv-backend=c10d
  • Auto-requeue on preemption — checkpoint-on-signal, job re-queues and resumes

Multi-Dimensional Parallelism  In Progress
Combining Data, Tensor & Pipeline Parallelism via DeviceMesh — the path to training 1T+ parameter models.
  • Tensor Parallelism (TP) — split weight matrices across GPUs within a node; each rank owns a row or column shard
  • Pipeline Parallelism (PP) — slice model by layers across stages; only activations cross boundaries (works over slow inter-node links)
  • DeviceMesh (DP × TP × PP) — named grid dimensions give each strategy its own process group; used by Megatron-LM & GPT-4
  • 4D / 6D parallelism — add Context Parallel + Expert Parallel for MoE at full scale
