📚 Getting Started with OMNI-Train

🚀 What is OMNI-Train?

OMNI-Train is a unified framework for fine-tuning machine learning models with distributed training support. Whether you're training image classifiers, large language models, vision-language models, object detectors, or embedding models, OMNI-Train provides a consistent interface with powerful features like LoRA, quantization, and multi-GPU training.

🖼️ Image Classification (CNN)

Fine-tune ResNet, ViT, EfficientNet, ConvNeXt, and more for image classification tasks.

💬 Large Language Models (LLM)

Train or fine-tune models like LLaMA, Mistral, Phi with full fine-tuning or LoRA.

👁️ Vision-Language Models (VLM)

Fine-tune multimodal models like LLaVA, BLIP-2, Qwen-VL for image understanding.

📦 Object Detection

Train YOLOv8/v11 models for real-time object detection with YOLO or COCO datasets.

🔗 Embedding Models

Fine-tune sentence-transformers, BERT, CLIP for semantic similarity and retrieval.

⚡ Distributed Training Strategies

OMNI-Train supports three distributed training strategies. Choose based on your hardware and model size:

None (Single GPU)

1 GPU

Simple single-GPU training. Best for smaller models or when you have limited hardware.

  • No setup required
  • Works with any GPU
  • Limited by single GPU memory

DDP (Data Parallel)

Multi-GPU

Distributes data across GPUs. Each GPU holds a full model copy. Fastest for models that fit in memory.

  • Near-linear speedup
  • Model must fit on 1 GPU
  • Effective batch = batch x GPUs

FSDP (Model Parallel)

Large Models

Shards model parameters across GPUs. Enables training models larger than single GPU memory.

  • Train 7B+ models on consumer GPUs
  • Memory efficient
  • Slightly more overhead

When to Use Each Strategy

Scenario Recommended Strategy Why
Small model (ResNet, BERT-base) none or ddp Model fits easily in memory; DDP gives speed boost
Medium model (LLaMA-7B with LoRA) ddp LoRA reduces memory; DDP maximizes throughput
Large model (LLaMA-13B+ full fine-tune) fsdp Model sharding required to fit in memory
Limited VRAM (8GB or less) none + quantization 4-bit quantization reduces memory 4x

📝 YAML Configuration Examples

OMNI-Train uses YAML files for configuration. Here are examples for different model types:

LLM Fine-tuning with LoRA

# Fine-tune LLaMA-2 with LoRA on a custom dataset
seed: 42

model:
  type: llm
  name: meta-llama/Llama-2-7b-hf
  finetune_mode: lora
  quantize: true
  quant_bits: 4
  lora_r: 16
  lora_alpha: 32

data:
  type: hf_dataset
  name: tatsu-lab/alpaca
  max_seq_len: 2048

distributed:
  strategy: ddp
  mixed_precision: true

training:
  epochs: 3
  batch_size: 4
  lr: 2e-4
  grad_accum_steps: 8

Image Classification (CNN)

# Fine-tune ResNet50 for custom image classification
model:
  type: cnn
  name: resnet50
  num_classes: 10
  pretrained: true
  freeze_backbone: false

data:
  type: image_folder
  name: ./data/my_images
  image_size: 224

distributed:
  strategy: ddp
  mixed_precision: true

training:
  epochs: 20
  batch_size: 32
  lr: 1e-3

🎯 Key Concepts

LoRA (Low-Rank Adaptation)

LoRA is a parameter-efficient fine-tuning method that adds small trainable matrices to the model instead of updating all parameters. This dramatically reduces memory usage (often 10-100x) and training time while achieving similar performance to full fine-tuning.

  • lora_r: Rank of LoRA matrices. Higher = more capacity but more memory. Typical: 8-64
  • lora_alpha: Scaling factor. Usually set to 2x lora_r
  • lora_target_modules: Which layers to apply LoRA (e.g., q_proj, v_proj)

Quantization

Quantization reduces model precision from 32-bit to 4-bit or 8-bit, dramatically reducing memory usage. Combined with LoRA, you can fine-tune a 7B parameter model on a single 24GB GPU.

  • 4-bit: Maximum memory savings (~4x reduction)
  • 8-bit: Better quality, moderate savings (~2x reduction)
  • Not compatible with FSDP (use DDP or single GPU)

Gradient Accumulation

When you can't fit a large batch in memory, gradient accumulation simulates larger batches by accumulating gradients over multiple forward passes before updating weights.

  • Effective batch size = batch_size x grad_accum_steps x num_gpus
  • Example: batch_size=4, grad_accum=8, 2 GPUs = effective batch of 64

Tips for Training

  • Learning rate: Lower LR for LoRA (1e-4 to 2e-4) vs full fine-tuning (1e-5 to 5e-5)
  • Batch size: Larger effective batch sizes generally improve training stability
  • Checkpoints: Save checkpoints regularly to resume if training crashes
  • Warmup: Use warmup steps (100-1000) to stabilize early training
  • Gradient clipping: Keep grad_clip around 1.0 to prevent exploding gradients
